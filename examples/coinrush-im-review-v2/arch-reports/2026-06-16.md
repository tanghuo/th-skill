## 3. 聚焦评审发现（维度 A）

### A1. 新增可持久化 Stream 派发链路：已建成但默认休眠，是首个落地的多 gateway 基础设施

- **现状**：dispatch 传输三态——`pubsub`（legacy，默认）/ `shadow`（双写测量）/ `stream`（durable）。配置默认 `TransportMode=pubsub`、`ShadowMode=false`（证据：强，`internal/config/config.go:142,146`；`internal/store/redisstore/dispatch_publisher.go:61-66`）。生产当前走 legacy pub/sub，Stream 发布/消费休眠在开关后。NodeRegistry + 心跳则**无条件运行**（`apps/gateway/main.go:34 go ws.RunNodeHeartbeat`，不受 mode 门控），即默认模式下节点表也一直被填充。
- **判断**：这是基线后状态拓扑的实质变化——基线记的是"per-NodeID Redis pub/sub fanout"，现在多了一套 durable Stream + 节点注册作为旁路。它是 IM 项目第一处具体的多 gateway 基础设施，落在单实例（1×1）现实里。
- **最强反方（对"是否过度建设"）**：over-implementation 审计（06-15）已验证其单实例价值——ActiveNodes 空时返回 ErrNoActiveDispatchNodes，把"NodeID 配错→写到无人消费 stream→静默丢"换成显式失败，且 NodeID 变更自愈；durable+去重+pending 恢复是可靠性内核非投机扩面。心跳常驻成本极低。**所以这不是"现在该砍"或"现在该启用"。**

- **方向**：链路本体不动。但需进账本登记为新决策条目并跟踪 cutover（见 A2 + 第 5 节），否则状态拓扑的"双轨"事实会在下期评审里被当新发现重复分析。

# 维度 C 增补（同日第三次评审，2026-06-16）

## C 维度发现

### C1. 进程内 ticker 选型维持；本期补 per-round deadline 关闭"单轮挂住停调度"，但统一 1min 默认无 per-job 逃生口，对长结算 job 是潜在失败模式

- **现状**：`699c4937` 给 ticker 加 `RunTimeout`（`ticker.go:126-135`，`context.WithTimeout`，未设时默认 `minTickerRunTimeout=1min`）。逐项核对全 15 个 ticker——**无一覆盖 RunTimeout**（`grep 'RunTimeout:'` 空），含月度/日结算 job 全吃 1min 默认（证据：强，main.go:45 tickers 切片 + ticker.go:133-135）。
- **判断**：正面——基线 C1 / resilience 06-15 P2 提的"job 给根 ctx 无 deadline、单次挂住停调度"失败模式已关闭，超时单轮被杀并打 `status=timeout` 日志，下一轮 tick 继续。待观察——结算 job 若单轮耗时逼近/超过 1min，会被反复杀在同一点；是否永久停滞取决于结算是否增量提交（增量则下轮续跑安全，单事务全或无则停滞）。
- **最强反方**：当前 1 人项目 + 适中主播量，1min 对结算绰绰有余；结算幂等可重跑（RowsAffected 语义，money 审计 06-12 已证），超时被杀可恢复非数据损失；现在给每个 job 配 RunTimeout 是为想象规模提前付配置（违反 config-vs-const 克制）。
- **结论**：维持现状。条件进账本——结算 job 单轮耗时接近 1min（届时临时打点/日志观测）或主播基数显著增长时，给结算类 job 设更大的 per-job RunTimeout。

### C2. Redis Stream 作为 durable 派发传输：失败模式逐项已处理，维持；maxlen/replay 窗口是 scale-coupled 常量

- **现状**：派发 durable 走 per-node Redis Stream（`dispatch_stream.go`）：approx `MaxLen=10000` 裁剪（覆盖 ~5min 回放窗口）、consumer group + `XReadGroup`/`XAck`、`XAutoClaim` 恢复 idle PEL、`replay-age` 30s 跳过陈旧 display 效果、按 `requestId` 去重（`DispatchDedupeTTL`）。失败模式逐项：
  - 丢失（trim）：消费者宕机超 ~5min backlog → 最旧条目被裁。display 由 replay-age 主动跳过；state 由对账 job（flushroomstate 等）从 MySQL 收敛——dispatch 丢失 ≠ 状态永久丢失。
  - 重复：consumer group at-least-once + requestId 去重（display）+ 收敛安全（state 不去重，重投无害，`dispatch_stream_consumer_test.go:411` 印证）。
  - PEL 卡住：`XAutoClaim` 恢复 → 已处理。
  - 消费者组 rebalance：每 NodeID 一条独立 stream、单消费者，无多消费者 rebalance 复杂度。
- **判断**：选型对当前量级恰当，durable 内核的失败模式都有对应处理。`MaxLen=10000`/~5min 是 scale-coupled 常量——吞吐增长时回放窗口缩短。
- **最强反方**：是否为 multi-gateway 想象规模提前上的复杂方案？over-impl 审计（06-15）已验证单实例价值（配错 NodeID → 显式失败而非静默丢），非过度。
- **结论**：维持现状。条件进账本——派发吞吐显著增长或 multi-gateway 启用时，重估 `DispatchStreamMaxLen` 与回放窗口。

### C3. 消息传输选型一致性：仓库无外部 MQ，3 个"未来 MQ consumer"是无传输休眠 stub；激活时应默认 Redis Stream 防选型漂移
