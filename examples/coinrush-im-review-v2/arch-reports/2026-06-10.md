## 3. 聚焦评审发现

### A. 运行时与状态

**A1. 15 个进程内 ticker 中仅 6 个有分布式锁，多实例部署即重复执行**
- 现状：main.go:38-55 注册 15 个 ticker；含锁的仅 anchorearning/anchorconversionroi/anchorrevenueboost/flushroomstate/presencecleanup/stalepresencereconcile 六个（证据：强，grep AcquireLock/Lock 文件清单）。gift reconcile、anchor live settle、room notice send 等无锁。
- 判断：单实例部署下现状正确；多实例时无锁 job 重复执行，钱相关的 anchor live settle 依赖幂等语义兜底。
- 最强反方：当前就是单实例部署，给每个 job 加锁是为想象中的规模提前付复杂度；且结算重跑幂等是仓库已有纪律（RowsAffected 语义）。
- 方向：不动。触发信号进账本（多实例立项 → 全量 job 锁/幂等审计）。

**A2. roomactor 多实例策略是 takeover（抢所有权）而非转发，AdvertiseAddr 是预留死字段**
- 现状：请求落在非 owner 节点时直接 `TakeoverRemoteOwner` 抢走所有权（证据：强，`logic/process_signal_router.go:34`、`process_signal_membership.go:194`、`checkrongmessagecallbacklogic.go:214`）；`AdvertiseAddr` 被写进 lease（actor_manager.go:235）但全仓无读取方。
- 判断：多实例下两节点会就同一房间互相抢 owner（ping-pong），actor 串行化退化为靠 Redis lease CAS 兜底。这是水平扩展的第一个断点。
- 最强反方：takeover 比节点间转发 RPC 简单一个量级；单实例下零影响；真到多实例时若 LB 做房间亲和，ping-pong 频率可控。
- 方向：不动，不实现转发。进账本：多实例后观测 takeover 频率再决定。

**A3.（正面确认）状态收敛链路结构健康**
- MySQL 事实源 + Redis 投影 + flushroomstate/stalepresencereconcile/emptyroomcleanup 对账兜底，与既定语义一致；gateway WS fanout 走 per-NodeID Redis pub/sub（handler.go:227-247），多 gateway 实例已天然支持。ticker 有 panic recover + metrics（ticker.go:79-103），排障有基本观测面。无需调整。

### B. 目录与职责

**B1. logic 层双形态：35 个文件是薄适配，22 个仍承载业务**
- 现状：残留业务集中在 room-core 信令处理（`checkrongmessagecallbacklogic.go` 618 行、`process_signal_gift.go` 475 行、`process_signal_membership.go` 463 行）（证据：强，wc -l + grep 委托统计）。logic 也是全仓 fix 密度第一。
- 判断：这是仓库最大的结构债，但 CLAUDE.md 已认账为 backlog，且 process_signal_* 与 roomactor/信令路由紧耦合，迁移成本显著高于普通 feature。
- 最强反方：room-core 是改动最频繁的区域，半途强迁会让每个功能 PR 都踩迁移雷区；现有"新功能落 application、存量缓迁"策略代价最低。
- 方向：维持渐进策略，不为本次评审造重构任务。守住"新增不委托文件 = 违例"这条线（进账本触发信号）。

**B2. owner resolution 模式散落 8+ 个 logic 文件**
- 现状：`ResolveForMutation` + 非本地则 `TakeoverRemoteOwner` 的样板在 process_signal_router、membership、checkrongmessagecallback、closeroom、updateroom、acceptroomadmininvite、removeroomadmin 等处重复（证据：强，grep 调用点）。
- 判断：横切关注点在适配层重复，但模式简单、行为一致。
- 最强反方：抽 helper 只省 5 行且多一层间接；等 room-core 迁入 application 时自然收口一次到位。
- 方向：不单独动，归入 room-core 迁移时一并收口。

**B3. `pkg/` 已空，只剩空目录**
- 现状：`pkg/bizcode` 已迁 `internal/bizcode`，`pkg/conv` 目录无任何文件（证据：强，find pkg -type f 为空）。
- 判断：空壳目录会误导"共享代码可以放这"。
- 方向：删除 `pkg/`，一行 chore。

### C. 选型失败模式

**C1. 进程内 ticker 扛钱相关管线（结算/工资/ROI）：维持现状**
- 失败模式逐项过：重复执行 → 锁 + RowsAffected 幂等已覆盖；进程重启丢一轮 → 间隔重跑可补（管线按 stat_date 重入）；无重试 → 下一轮 tick 即重试。真实痛点未出现（fix 集中在口径/裁剪逻辑，非调度形态——job 的 46 个 fix-touch 抽样为业务口径修正）。
- 最强反方（对"维持"）：结算在业务高峰期与在线 RPC 同进程争资源。但无延迟受损证据，属想象中的问题。
- 结论：维持现状；"多实例立项"为再评估条件（已进账本）。

**C2. MySQL 5.7 + Redis 双存储扛全部状态：维持现状**
- Redis 丢投影/漂移的失败模式已有 MySQL 驱动的对账 job 兜底（既定设计）；MySQL 5.7 语法约束已写进 CLAUDE.md 防误用。无已发生痛点，不引入新组件。
