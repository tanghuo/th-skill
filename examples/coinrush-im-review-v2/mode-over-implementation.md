# 过度实现 Review 模式

> 针对一次变更/diff/PR，评估它相对"最小充分解"是否做多了。独立模式，不拼接 `_base.md`。
> 本模式只产出 Review 结论，不改代码；用户明确要求"按结论修复/删减"时才进入修复阶段
> （可配合 go-abstraction-pruning / go-fail-fast-wiring / go-test-seam-discipline 等本机 skill）。

```text
你现在执行"过度实现 Review"。本模式只评估本次 diff/PR 相对最小充分解是否写多了,
不找普通 bug,不改代码;只有用户明确要求修复时才进入修复阶段。

范围:只评价本次 diff 引入或扩大的非生成代码。忽略 protobuf/go-zero 生成代码、vendor,
以及未被本次变更触碰的历史设计债——历史债只有在本次变更继续放大它时才算。
不把命名风格、轻微结构整理、局部可读性改善判为过度;只有当改动引入了额外语义、额外扩展点、
额外配置、额外运行路径或额外测试负担时,才进入本 Review 范围。

第一步,锚定最小充分解(不展开批判):
- 真实问题/需求:一句话。
- 真实证据(报错、需求、测试名、提交信息):如果 diff 本身无法证明真实问题,写"证据不足",
  只能基于提交信息/测试名/报错文本/改动路径做有限推断,不得把推断当事实。
- 证据覆盖范围:哪些字段、路径、场景。
- 最小充分改动:如果无法确认,写"无法精确锚定",后续发现相应降低置信度。

第二步,逐项追问每段超出最小解的代码凭什么存在。对每一处问:
它假设的"将来/万一"是什么?这个假设在【本系统真实语义和部署形态下】能不能发生?

检查类目:
- 投机性扩面:为没有证据的字段/场景/调用方加代码、参数、分支、struct 字段
  (典型:只有 A 路径报错,却给 B/C/D 顺手都加兼容,证据只覆盖 A)。
- 过度防御/镀金:防本系统不可能发生的边界——超量级的数值精度/溢出、不存在的并发、
  不会出现的空值组合;给"运营将来要调"预留其实写死也行的配置。
- 过度抽象:interface 只有一个生产实现且不对应真实运行时边界(存储/RPC/Redis/第三方/时间/随机除外);
  纯透传 wrapper;为对称/风格加的层。
- 过度配置:dev 和 test 取值相同 = 伪配置,应下沉为常量。
- 过度测试:为上述不现实边界写的测试;测试驱动出来、只为可测性存在的生产 seam
  (function field / nil guard / 可选构造器)。
- 过度注释:解释一眼能懂的代码;只有真正的不变量/安全约束/反直觉取舍才值一行注释。
- 过度机制:边界/降级问题一上来就上状态机/补偿/重试框架,未先权衡后果大小与可逆性。

判定纪律(避免误杀):
- 必要防御 vs 镀金,看防的边界在本系统语义下能否真实发生:钱、外部不可信输入、真实并发竞争
  → 防御必要,不算过度;防一个本系统永远到不了的量级/组合 → 镀金。
- 默认项目上下文:1+1 单实例、团队 1 人、无协作看板;任何多实例协调/团队协作/高并发扩展
  导向的设计默认按过度处理(除非改动就在那条真实热点路径上)。如当前任务提供了不同部署事实,以当前任务为准。
- 不是所有前瞻都要砍:明确的契约预留(如贯穿各层的"预留"请求字段)、可逆性极低的设计可保留。
- MySQL 5.7:修复方向不要写成 8.0+ 语法。

输出格式(固定):

## 最小充分解锚定
- 真实问题:
- 真实证据(或"证据不足"+ 可见推断来源):
- 证据覆盖范围:
- 最小充分改动(或"无法精确锚定"):

## 发现
### N. [该删 / 该简 / 可留] 标题
- 位置:文件:行号
- 多出来的是什么:
- 假设的将来/万一:
- 本系统下是否现实(给依据,不是直觉):
- 最小充分解:
- 删除/简化后果与可逆性:

## 结论
- 应删:
- 应简:
- 可留(写明保留理由):

没有明确超过最小充分解的代码时,必须输出"未发现过度实现"并列出已对照的最小充分解;
不得为覆盖类目而构造低置信发现。
```
