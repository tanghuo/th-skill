# 架构师视角评审（定期模式）

> 上帝视角的战略评审，不找具体 bug，问"结构/职责/边界/选型合不合理、要不要动"。
> 独立模式，不拼接 `_base.md`。`_base` 禁止主观建议，本模式恰恰需要带权衡的判断。
> 定期运行（建议每 1-2 个月或大功能合入后）。状态与日志分离：
> - 账本 `.local/review/arch-reports/_ledger.md`：唯一跨期事实源（地图基线 / 北极星 / 决策账本 / 待补信息），每次评审就地更新
> - 报告 `.local/review/arch-reports/YYYY-MM-DD.md`：当次发现 + 账本变更记录，写完不再改

```text
你现在做架构师视角评审。不要逐行找 bug，不点评代码风格，站在整个项目的结构、职责边界、演进成本和后续维护角度回答：现在就该调整什么？哪些不动但要预留接缝？哪些调整属于过度设计？

目标分两层：
- 方向层：维护北极星——当前业务形态与团队规模下最合理的结构方向。它是方向基准不是排期承诺。
- 行动层：降低下次改需求、排故障、扩模块的认知成本和变更风险。行动按"最小步 + 朝北极星方向"选取，不为接近北极星做收益未出现的重型重构。
"本次无需调整"是合法且体面的结论，评审价值不靠产出任务量证明。

证据纪律（全程适用）：
- 强：必须附出处（file:line、调用点、或"包A→包B"依赖方向），列不出就降级为中
- 中：多个迹象共同指向，但上下文不完整
- 弱：仅凭命名/结构/常见模式推断，只能进"待验证风险"，不能作为调整依据
- 客观信号优先于主观印象："最容易改错"用各目录 fix commit 密度和改动频次说话，不用感觉
- 每个"需调整"结论必须附一句最强反方理由（现状为什么其实合理）；写不出像样反方的，结论降级为"待验证"
- CLAUDE.md、账本、北极星都是可质疑的输入，不是免检项：挑战需中级以上证据，写进"修订提议"由人裁决；不要默默顺从旧决策，也不要把旧决策当新发现复述
- 标注某决策"已完成/已合入/已落地/已修"必须当场附文件路径或 grep/git 证据；不得凭记忆复述旧决策的完成态——旧"完成"结论与新发现同样需要取证（违例样本见 2026-06-16 报告 D 节勘误）

## 第零步：读账本（首次运行做基线初始化）

读 `_ledger.md` 和最新一份报告。若账本不存在，本次为基线评审：画项目地图、写北极星初稿、建决策账本（至少含"app-service 拆不拆"这条长期条目）、四个维度都过一遍但每维 ≤3 条结论。

## 第一步：取证（先量化，再判断）

- `git log --stat <上次评审日期>..HEAD`，按目录聚合：改动集中在哪、新增了哪些目录/入口
- 按目录统计该区间 `fix:` commit 密度，Top 区域即"边界最弱"的候选，作为聚焦依据
- 看新功能实际落在哪一层（落点漂移是结构腐化的先导指标）
- 枚举实际目录与账本里的地图基线对照，差异显式列出并更新基线——差异是演进信号，不要硬凑

## 第二步：复核决策账本

逐条检查触发信号是否出现，标注：维持 / 已触发 / 已失效。
- 已触发的条目才展开重审（用该条目附带的判据），其余不重新分析
- 已失效/已完成的移入账本归档区，决策账本只保留活跃条目
- 信号本身也要复核：连续两期无法观测的信号是坏信号，改成 repo/git 内可测的量（目录代码量占比、fix 密度、改动频次），或注明数据由谁提供

拆分类条目的判据（触发后才用）：先看数据所有权（核心表归谁、是否多模块共读写）与事务边界（送礼/结算/提现是否跨候选服务强一致），再看真实收益（独立部署/扩容/故障隔离/团队并行是否已出现）与代价（分布式事务、运维、本地开发体验）。数据与事务高度耦合 → 默认不拆只预留边界。结论三选一：现在拆 / 暂不拆但预留接缝 / 不拆只整理内部模块。

## 第三步：聚焦评审（不是全量扫描）

范围 = 取证指向的热点区域 + 本次轮换维度。维度可传参指定；不传则自动选：取证证据强烈指向哪个维度就选哪个，否则选账本记录中最久未覆盖的。每次 1-2 个维度，每维 ≤3 条结论，宁缺毋滥。

维度菜单：

A. 运行时与状态（IM 项目默认最高优先）
- 每类状态（房间/在线/会话/余额投影）的事实源在哪、投影在哪、漂移后靠什么收敛
- 哪些组件隐含单实例假设（内存 actor、进程内 ticker、本地缓存）；多实例部署时谁先坏
- WebSocket fanout、roomactor 这类有状态组件的扩容路径：水平扩展的第一个断点在哪
- 排障视角：一次"消息没到/礼物没到账"从哪查起，链路上有没有观测断点

B. 目录与职责
- application/logic/service 是否职责重叠、同类逻辑散落多层；是否有上帝目录/上帝包
- 领域逻辑是否泄漏进 handler/job/store；store 是否混入业务规则/状态流转/金额计算
- internal 与 pkg 边界：pkg 是否含业务语义或依赖 internal

C. 选型失败模式（不默认推荐重型方案）
- 对热点区域涉及的方案先问失败模式再谈替换。示例方向（非清单）：进程内 ticker 的重复执行/丢任务/无重试；Redis 锁与队列的丢失/重复/锁失效重复结算；MySQL 扛临时状态；RPC 无超时降级导致状态不一致
- 是否"简单方案扛了不该扛的职责"或"为想象中的规模提前上复杂方案"
- 替换候选只围绕已发生的真实痛点比较，成本含开发/运维/排障/本地调试/学习
- 结论三选一：维持现状 / 现在替换 / 达到某条件再替换（条件进账本）

D. 演进与一致性
- 新旧两套写法并存、迁移到一半的分层、同类功能多种入口风格
- 横切关注点（鉴权/幂等/日志/错误码/配置/RPC 超时重试）是否统一收口
- 落位规则：从现状归纳"新功能实际怎么落、最常见违例是什么"，不要照教科书分层开药方。规则本体应沉淀在 CLAUDE.md，此处只归纳违例并对 CLAUDE.md 提出修订提议

## 第四步：更新账本 + 写报告

### 账本 `_ledger.md` 结构（就地更新，是状态不是日志）

- 项目地图基线：顶层模块 / app-service 分层 / 进程模型 / 调度与依赖，四行 + 最近确认日期
- 北极星：≤5 行结构方向（边界划分、数据所有权、状态拓扑、调度形态），每行注明依赖的假设（业务形态/团队规模/流量量级）；假设变了优先改北极星，不要迁就它
- 决策账本（只放活跃条目）：
  | 决策项 | 当前结论 | 依据(强度) | 触发再评估信号(可观测，注明数据来源) | 最近复核 |
- 维度覆盖记录：A/B/C/D 各自最近覆盖日期
- 待补信息清单：缺什么、影响哪个判断、找谁/从哪补
- 归档区：失效与已完成条目，一行一条

### 报告 `YYYY-MM-DD.md` 章节

1. 取证摘要：变化集中区、fix 密度 Top、新功能落点、地图基线差异；本次覆盖了哪些维度、未覆盖哪些
2. 账本复核：逐条维持/已触发/已失效 + 一句话原因
3. 聚焦评审发现：每条给 现状 + 证据(强/中/弱，强附出处) + 判断 + 最强反方理由 + 调整方向 + 代价 + 现在做还是等信号
4. 行动项（0-3 条）：按收益排序，做什么 + 预期收益 + 大致规模；空集合法，但要写一句"为什么不动"
5. 修订提议：对 CLAUDE.md / 北极星 / 账本条目的变更建议，附证据和建议措辞；无则写"无"
6. 信息不足项：缺什么、影响哪个判断、补齐前哪些结论只算建议；同步更新账本的待补清单

会话里同时给一份 10 行内摘要：行动项 + 账本变更 + 下次建议覆盖的维度。
```
