# 错误吞噬 & nil/零值边界

> 合并原"错误处理黑洞"与"边界条件&nil"：都是"异常输入/异常结果被当成正常值继续用"。

```text
审计错误被吞掉后继续执行、以及 nil/零值未检查直接使用的问题。

起点：
- logic / job 中 `_ =`、err 被 log 后继续执行、err 被覆盖的位置
- RPC 返回值与 Redis 查询结果的直接使用点（外部用户信息、房间查询、余额查询）
- JSON 反序列化后的字段访问；map 取值后零值直接参与计算

错误吞噬：
- error 被 log 后继续执行而非 return/rollback/补偿
- 事务、Redis、RPC、第三方 API 失败后仍使用其结果
- Commit/Rollback/Close/Flush 失败影响一致性却被忽略
- recover 后吞掉错误，调用方误以为成功
- 误报过滤：明确 best-effort 且后续不依赖该结果的，不报告

nil/零值边界：
- map 取值零值无法区分"不存在"与"值为零"——金额、计数、状态、权限的零值语义是否危险
- 类型断言未检查 ok；slice 下标越界；指针字段直接解引用
- RPC/Redis 返回 nil 或空结构后字段直接使用
- 可选字段、历史数据缺失字段、新增字段的零值路径
- nil guard 掩盖必需依赖缺失导致静默降级（必需依赖应启动时 fail-fast）

只报告后续逻辑依赖该结果、或会导致 panic、金额错误、权限错误、状态错误的问题。每个发现说明：错误/零值在哪产生、为什么被当正常值、触发后果、最小修复方向。
```
