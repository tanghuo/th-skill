# 反事实最小解审查 (counterfactual-baseline)

> 不找普通 bug,而是对每个治理/修复机制强制构造"更短等价候选"并做失败矩阵对比,
> 只在当前实现被某个更短候选"支配"(同一安全不变量下、所有关键失败场景不更差、成本更低)时才报。
> 拼接 `_base.md` 执行。原"过度兜底 / 僵尸机制"是本专题"删除候选支配"的特例,已并入本专题。

```text
你现在做"反事实最小解审查"。不找普通 bug,只回答一个问题:
项目里现有的治理/修复机制,是否被一个更短的等价方案支配
(同一安全不变量下、失败矩阵不更差、成本更低)。

核心方法:不凭空想"任意更优解",而是从高机制密度点出发,机械地构造更短候选并做支配证明。
能报的只有"被支配的劣化实现",不能报"我想象中更好的架构"。

范围与预算纪律(本主题天然宽,严禁只 grep 不深读、严禁假装全库穷尽):
- 候选池(两路汇合,不止 entry-map):① entry-map.md 的 job / 双写 / 状态机 / 对账入口作第一路种子;
  ② 再 grep 关键词补一轮——reconcile|retry|fallback|pending|outbox|dirty|compensate|ensure|restore|flush,
  以及 helper / repo 内部方法 / 状态字段 / 注释里的治理信号(entry-map 不覆盖这些)。
- 从候选池选 3-5 个深挖,并说明本轮为何选中这几个(机制密度 / 近期改动 / 可疑度);逐个走完五问 + 失败矩阵;
  候选池其余项只列入"覆盖范围",不得暗示已全库穷尽。

抓手(高机制密度点):
- 补偿 / 对账 / 重试 / outbox / pending / fallback / 双写 / 二次校验 / cache 回填
- "先写 A 再写 B / 失败后修复 / 失败后重试"的多步流程
- 为某个异常窗口新增的状态、字段、job、接口
- 注释里出现"保证一致性 / 防止丢失 / 收敛 / 容错 / 兜底"的位置
- 命中"常见待核验前提"的机制(这些前提只在部分链路成立,须逐条核验,不得默认全仓为真):
  未上线、1+1 单实例、DB 权威、actor 串行、已有热路径复查

对每个机制,固定五问:

1. 目标不变量
   不是"修 Redis/DB 不一致",而是真正要守的安全不变量:
   不能越权 / 最终向权威源收敛 / 扣费后必须发货 / 不重复结算。
   候选必须满足同一条不变量,不得偷偷弱化。

2. 现有权威源与局部前提(逐条核验,不得照搬其他链路的结论)
   权威源:DB / Redis / actor 内存 / 外部钱包 / 状态机,谁才决定最终生效。
   局部前提:本机制所在链路是否 actor 串行?是否已有热路径复查?是否向单一权威源收敛?
   这些前提只在部分链路成立(房管链路成立,礼物/钱包/结算/外部券未必),逐条给代码证据;
   未证实的前提不得进入支配证明。不先确定权威源,后面的失败矩阵无意义。

3. 当前实现的失败矩阵(先识别真实失败维度,不套模板)
   先找真实副作用边界:本机制有几个会失败的写入/副作用点?
   - 两步及以上写(双写 / 写库+发外部 / 多副作用链)才填 A 成功 B 失败 / A 失败 B 成功。
   - 纯 guard / 单点状态机 / 定时对账 / 缓存回填没有 A/B 两步,不要硬套;不适用维度写明"不适用:原因",
     并替换成该机制真实失败维度(对账:漏扫 / 重复扫 / 扫到脏候选;缓存回填:回填过期值 / 击穿 / 与源不一致;
     guard:误放行 / 误拦截)。
   通用维度(适用才填):重试 / 重复调用 / 超时;并发与串行边界(若该链路确为 actor 串行);
   重启 / 冷启动(从权威源重建——劣化常在这一维暴露)。
   每格写:落到什么状态、是否 fail-closed、是否向权威源收敛。

4. 构造至少 3 个更短候选(从下列选,不凭空发明):
   删除机制 / 改注释+测试锁语义 / 复用已有收敛路径 /
   检查前移或后移 / 改 fail-fast / 收敛到单点权威。

5. 支配判断
   每个候选重填同一张失败矩阵:
   - 所有关键场景"不更差"且成本更低 → 当前实现是劣化实现,报告。
   - 部分场景更好、另一些更差 → 不是支配,不报(可作权衡记录,不算 finding)。

判定纪律(防止退化成主观架构幻想):
- 支配证明必须基于已读代码的事实(权威源是谁、热路径查不查、是否 actor 串行、
  重建从哪恢复),不得基于"假设运营会怎样 / 将来可能"。
- 候选不得弱化目标不变量来显得更短(把"最终一致"降成"尽力而为"是降级,不是支配)。
- 重构成本评估:仅当本次审计已确认该模块未上线 / 用户明确声明可破坏性重构 / 无生产数据兼容约束时,
  成本才按低处理;否则按当前部署事实评估。资金、外部合同、历史数据迁移路径即使未上线也可能高成本。
  不得用"改动大"为保留劣化实现辩护,也不得默认所有改动都低成本。
- 没有失败矩阵 + 支配证明的,一律降为"观察",不得报为问题。
- MySQL 5.7:候选方向不要写成 8.0+ 语法。

与邻近专题边界:
- consistency-state:审"状态本身是否漂移";本专题审"为治理漂移而加的机制是否被更短解支配"。
- mode-over-implementation:审一次 diff 是否做多了;本专题针对全库存量,且主动构造对照物。

附加输出字段(在 _base 发现格式上,每条 finding 必须补全;缺任一项即降为"观察"):
- 当前机制(位置:文件:行号)
- 声称目标(真实不变量)
- 权威源
- 当前失败矩阵
- 候选集(至少 3 个具体候选,来自第 4 问;不是"应该更简单")
- 被淘汰候选及原因(逐个:在哪个失败场景更差 / 不满足不变量)
- 胜出候选(候选集中最强且最稳的那个)
- 支配证明(胜出候选失败矩阵逐场景不更差 + 成本更低的依据)
- 最小处置

非发现的沉淀(写入 _base 输出的"未发现问题的检查项",防止下轮重推同一机制):
- 已深挖但不报的机制,逐个列:机制 → 构造了哪些候选 → 胜出候选为何未支配
  (在哪个失败场景更差 / 不满足不变量)或当前实现为何可留。
```
