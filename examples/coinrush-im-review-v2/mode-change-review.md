# 变更 Review 模式

> 用于 PR / diff / 最近提交的增量审计。独立模式，不拼接 `_base.md`。

```text
你现在做变更 Review，只审计当前变更引入的问题，不审计历史遗留。

第一步，先输出变更清单（不分析，只列）：
- 新增/修改的入口与写路径
- 修改的状态字段、外部契约（api/proto）、配置项
- 修改的定时任务、callback、Redis key、schema/model
- 修改的权限判断、脚本、部署配置

第二步，审计。纪律：
- diff 只是索引，不是证据：每个改动点必须打开所在文件读完整上下文（函数全文、调用方、被调方），diff 视野外的破坏恰恰是变更 review 最该抓的
- 只报告本次变更直接引入或明显放大的问题；历史问题只有在本次变更会触发它时才报
- 重点核对：行为差异是否影响幂等、状态一致性、契约兼容、权限、结算口径
- 删除/重命名类变更要确认所有引用方（包括脚本、SQL、文档、Makefile）同步更新

严重度：P0=资损/泄漏/不可用  P1=重复结算/越权/不一致  P2=特定条件异常/语义错误  P3=低概率边界/技术债

每个发现说明：
- 为什么是本次变更引入或放大的（变更前后行为差异）
- 文件:行号 + 触发条件
- 最小修复方向

忽略测试文件、生成代码。没有发现就输出"未发现变更引入的问题"并列出检查范围——不要硬凑。
```
