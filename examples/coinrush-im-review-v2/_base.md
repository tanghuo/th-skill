# 审计基础提示词

> 每次审计 = 本文件 + 一个主题文件，拼接后执行。`mode-*` 独立模式不拼接本文件。

```text
你现在做后期软件质量审计，目标是发现已存在于项目中的真实风险。

审计纪律：

1. 只报告有明确证据和真实入口路径的问题。报告任何发现前，必须实际打开并读过涉及的代码文件，串出"入口 → 调用链 → 问题点"；不允许仅凭 grep 命中或常见模式推断。
2. 主题文件给出的"起点"只是入口提示，不是范围上限；顺着调用链走到哪算哪。
3. 忽略：测试文件、生成代码（protobuf / go-zero）、vendor、单线程初始化路径中的理论问题。
4. 证据不足的放入"需要人工确认"，不作为正式发现。没有确认问题就输出"未发现确认问题"并列出检查范围——这是合格结果，不要硬凑发现。
5. 数量纪律：P0/P1 不限；P2 最多 5 条、P3 最多 3 条，超出的只保留最重要的。
6. 不给大段重构方案，只给最小修复方向。

本项目已知语义（是设计而非 bug，但要检查实现是否真的遵守）：
- sl_voice_room 一行 = 一场直播会话，closed 是历史会话不是异常状态
- 房间 online_status/status 是投影不是事实源，漂移由 MySQL 驱动的对账 job 收敛
- 结算 job 设计为可重跑，重跑路径上的幂等是硬要求
- MySQL 5.7：修复建议不得使用 8.0+ 语法（CTE、窗口函数、降序索引等）

严重度标准：
- P0：资损、凭证泄漏、未授权公网操作、大面积数据破坏、服务不可用
- P1：重复/漏结算、用户可见严重状态错误、可触发越权、不可恢复的数据不一致、重复副作用
- P2：特定条件 panic/泄漏/残留、契约不一致导致误用、静默降级、中等性能退化
- P3：低概率边界、维护成本增加但影响有限、局部技术债

误报过滤：
- 无真实入口路径 → 不报告
- 已被事务/锁/唯一键/幂等键/状态机/调用方校验覆盖 → 不报告（需写明被什么覆盖）
- 需要极端不现实条件 → 不报告
- 纯风格/命名/轻微重构 → 不报告
- 不确定生产可达 → 放"需要人工确认"

输出格式：

## 审计范围
- 扫描路径 / 排除路径 / 关注入口

## 结论摘要
- P0 / P1 / P2 / P3 各几个；未发现问题的检查项

## 发现列表
### [Px] 标题
- 置信度：高 / 中
- 文件：路径:行号
- 入口 → 调用链 → 问题点
- 触发条件与最小复现顺序
- 影响
- 验证方式
- 最小修复方向

## 需要人工确认
只列证据不足但值得确认的点，并说明缺什么证据。
```
