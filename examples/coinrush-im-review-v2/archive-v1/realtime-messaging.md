# 实时消息 & 长连接语义

> IM / 语音房核心：连接生命周期、投递语义、在线状态。
> 与 `perf-capacity` 的"广播阻塞/无界队列"互补，本文件只看语义正确性，不看性能。

```text
扫描 WebSocket / 长连接入口、消息收发路径、在线状态维护和广播逻辑，检查实时语义是否正确。

连接生命周期：

- 建连鉴权失败后连接是否被正确关闭，残留连接是否仍能收发
- 断线 / 超时 / 主动关闭后，订阅、房间成员、在线计数是否被清理
- 心跳缺失或超时是否能检测到死连接，僵尸连接是否会一直占用资源
- 重连后会话状态、订阅关系、未读位点是否能正确恢复
- 多端登录 / 同账号多连接：踢线逻辑是否完整，被踢端是否真正断开
- 连接关闭与并发写是否存在 write on closed conn / 并发写同一连接

消息投递语义：

- 投递保证是 at-least-once 还是 at-most-once，是否与业务预期一致
- 消息是否携带去重键 / 序列号，客户端能否去重
- 同一会话消息顺序是否有保证，并发广播是否会乱序
- 离线消息是否落库 / 入离线队列，上线后是否补投，是否重复补投
- 已读回执 / ack 丢失后消息状态是否卡死或重复
- 单条消息广播给多接收方时，部分失败如何处理，是否半投递

在线状态 & 房间成员：

- 在线状态来源唯一，还是 Redis / 内存 / DB 多处维护可能漂移
- 进房 / 退房 / 掉线时成员列表和在线计数是否守恒（参考 consistency-state-machine 计数不变量）
- 房间关闭（live session 结束）后残留连接 / 订阅是否被清理

每个发现说明：

- 涉及的连接或消息路径
- 错误的语义假设 vs 实际行为
- 触发条件（断线 / 重连 / 多端 / 并发广播 / 部分失败）
- 用户可见后果（丢消息 / 重复消息 / 乱序 / 幽灵在线 / 卡死）
- 最小修复方向
```
