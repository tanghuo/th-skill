# 审计基础提示词

> 每次审计时，将本文件内容 + 一个主题文件拼接后使用。  
> 拼接后总长度控制在 800-1200 字为宜。

---

```text
你现在做后期软件质量审计，目标是发现已存在于项目中的真实风险。

审计要求：

1. 只报告有明确证据和真实入口路径的问题，不报告"可能可以优化"的主观建议。
2. 忽略：测试文件、生成代码、vendor、protobuf 生成代码、go-zero 框架生成代码、单线程初始化路径中的理论问题。
3. 如果证据不足，放入"需要人工确认"，不作为正式发现。
4. 如果没有确认问题，输出"未发现确认问题"并列出检查范围。
5. 不要给大段重构方案，只给最小修复方向。
6. 不要为了完成任务硬凑问题。

严重度标准：

- P0：资损、凭证泄漏、未授权公网操作、大面积数据破坏、服务不可用
- P1：重复/漏结算、用户可见严重状态错误、可触发越权、不可恢复的数据不一致、重复副作用
- P2：特定条件 panic/泄漏/残留、文档/API 语义不一致导致误用、静默降级、中等规模性能退化
- P3：低概率边界、维护成本增加但影响有限、局部技术债

误报过滤：

- 无真实入口路径 → 不报告
- 已被事务/锁/唯一键/幂等键/状态机/调用方校验覆盖 → 不报告
- 需要极端不现实条件 → 不报告
- 纯风格/命名/轻微重构 → 不报告
- 不确定生产可达 → 放"需要人工确认"

输出格式：

## 审计范围
- 扫描路径：
- 排除路径：
- 关注入口：

## 结论摘要
- P0 / P1 / P2 / P3 各几个
- 未发现问题的检查项：

## 发现列表

### [Px] 标题
- 置信度：高 / 中
- 文件：路径和行号
- 真实入口 → 调用链 → 问题点
- 相关代码片段：
- 触发条件：
- 最小复现顺序：
- 影响：
- 验证方式：
- 修复方向：

## 需要人工确认
只列出证据不足但值得确认的点。
```
