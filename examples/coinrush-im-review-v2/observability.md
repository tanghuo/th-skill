# 可观测性缺口

> 站在"线上出事时能不能定位"的角度，不是补 debug 日志。

```text
审计核心链路的日志、指标、告警是否足以定位线上问题。

起点：
- app-service 的 internal/metrics 现有指标清单——先看已经有什么，再找缺口
- 排障演练法：假设"礼物没到账""消息没收到""结算金额不对"三类工单，从日志能否还原链路；断点在哪一跳

重点：
- 关键动作（开关播、送礼、结算、回调、踢线、推送）是否有结构化日志，是否带 request_id / user_id / room_id 关联字段
- error log 是否缺关键上下文（哪个对象、哪步失败）
- 关键状态变化、幂等冲突、重复 callback、非法流转有没有指标
- 批处理跳过异常数据是否有计数或告警；关键异步任务失败是否无人知晓
- metrics label 高基数风险（user_id、token、完整 URL 入 label）

只报告影响线上定位、故障恢复或事故追踪的缺口。每个发现说明：哪条排障路径会断、断在哪、补什么（一条日志/一个指标/一个告警），不要求上全套 tracing。
```
